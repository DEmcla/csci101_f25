PART A: TRUE/FALSE (24 points total)

Each question is worth 2 points

Question 1. In an Entity Relationship Diagram (ERD), a rectangle represents an entity and an oval represents an attribute.

✓ TRUE - Rectangles = entities, ovals = attributes (standard ERD notation)

Question 2. A primary key can be composed of multiple columns working together (a composite key).

✓ TRUE - Composite keys are made of 2+ columns (e.g., {StudentID, CourseID})

Question 3. In crow's feet notation, a "crow's foot" symbol indicates the "many" side of a relationship.

✓ TRUE - Crow's foot (≺) indicates the "many" side

Question 4. First Normal Form (1NF) requires eliminating transitive dependencies.

✗ FALSE - 1NF eliminates non-atomic values and repeating groups. 3NF eliminates transitive dependencies.

Question 5. A functional dependency X → Y means "if you know X, you can determine Y."

✓ TRUE - This is the definition of functional dependency

Question 6. Bridge tables (junction tables) are used to resolve one-to-one relationships.

✗ FALSE - Bridge tables resolve many-to-many relationships, not one-to-one

Question 7. A partial dependency occurs when a non-key attribute depends on only part of a composite primary key.

✓ TRUE - This is the definition of partial dependency (violates 2NF)

Question 8. Second Normal Form (2NF) requires that the table must first be in 1NF and have no partial dependencies.

✓ TRUE - This is the definition of 2NF

Question 9. A foreign key in one table references the primary key of another table, creating a relationship between them.

✓ TRUE - This is the definition and purpose of foreign keys

Question 10. In a many-to-many relationship, you can directly connect two tables without a bridge entity.

✗ FALSE - Many-to-many requires a bridge entity/table in between

Question 11. Third Normal Form (3NF) is considered the industry standard for most database applications.

✓ TRUE - 3NF is the industry standard for most applications

Question 12. Update anomalies, insertion anomalies, and deletion anomalies are problems that normalization helps prevent.

✓ TRUE - These are the three anomalies normalization prevents

PART B: MULTIPLE CHOICE (40 points total)

Each question is worth 2.5 points

Question 13. Which of the following is an example of a non-atomic value that violates 1NF?

a) StudentID: 12345
b) PhoneNumber: "555-1234, 555-5678"
c) LastName: "Smith"
d) GPA: 3.75
ANSWER: b) PhoneNumber: "555-1234, 555-5678" - Multiple values in one cell violates atomicity (1NF)

Question 14. In crow's feet notation, which symbol combination represents a one-to-many relationship?

a) Single line on both ends
b) Single line on one end, crow's foot on the other
c) Crow's foot on both ends
d) Double line on both ends
ANSWER: b) Single line on one end, crow's foot on the other - One side = single, many side = crow's foot

Question 15. Consider a table ORDER_DETAIL with Primary Key: {OrderID, ProductID}. Given the functional dependency {OrderID, ProductID} → Quantity, what type of key is {OrderID, ProductID}?

a) Foreign key
b) Simple primary key
c) Composite primary key
d) Candidate key
ANSWER: c) Composite primary key - Made of two columns working together

Question 16. Which normal form specifically addresses the elimination of partial dependencies?

a) First Normal Form (1NF)
b) Second Normal Form (2NF)
c) Third Normal Form (3NF)
d) Boyce-Codd Normal Form (BCNF)
ANSWER: b) Second Normal Form (2NF) - 2NF specifically eliminates partial dependencies

Question 17. What is a transitive dependency?

a) When an attribute depends on the entire composite key
b) When A → B and B → C, so A → C indirectly
c) When an attribute depends on only part of the primary key
d) When two attributes depend on each other
ANSWER: b) When A → B and B → C, so A → C indirectly - This is the definition of transitive dependency

Question 18. In an ERD, what does a bridge entity (associative entity) represent?

a) A weak entity that cannot exist independently
b) An entity that resolves a many-to-many relationship
c) An entity with no attributes
d) The primary entity in a one-to-one relationship
ANSWER: b) An entity that resolves a many-to-many relationship - Bridge/associative entities break M:N into two 1:M relationships

Question 19. Consider this table: COURSE(CourseID, CourseName, InstructorID, InstructorName, InstructorOffice)

If CourseID → InstructorID and InstructorID → InstructorName, InstructorOffice, which normal form is violated?

a) 1NF
b) 2NF
c) 3NF
d) None - this is properly normalized
ANSWER: c) 3NF - Transitive dependency exists: CourseID → InstructorID → InstructorName/Office. Table is in 2NF (no partial dependencies with single-column PK) but violates 3NF

Question 20. What is the primary purpose of normalization?

a) To make queries run faster
b) To reduce data redundancy and prevent anomalies
c) To combine multiple tables into one
d) To create more complex database structures
ANSWER: b) To reduce data redundancy and prevent anomalies - This is the primary purpose of normalization

Question 21. In the functional dependency notation ISBN → BookTitle, which term describes ISBN?

a) Dependent
b) Determinant
c) Composite key
d) Foreign key
ANSWER: b) Determinant - The left side of X → Y is the determinant

Question 22. Which of the following scenarios would most likely benefit from denormalization?

a) A frequently-run report that requires joining 8 tables and is too slow
b) A brand new database with no performance issues
c) A table with update anomalies
d) A table that violates 1NF
ANSWER: a) A frequently-run report that requires joining 8 tables and is too slow - Performance issues are the main reason to denormalize

Question 23. Consider: ENROLLMENT(StudentID, CourseID, StudentName, Grade)

Dependency: StudentID → StudentName

What type of dependency problem exists?

a) Full dependency
b) Partial dependency

ANSWER: b) Partial dependency - StudentName depends on only part of the composite key {StudentID, CourseID}

Question 24. In crow's feet notation, what does a perpendicular line (|) on a relationship line indicate?

a) Optional participation (zero allowed)
b) Mandatory participation (at least one required)
c) Many instances
d) Exactly one instance
ANSWER: b) Mandatory participation (at least one required) - Perpendicular line = mandatory, circle = optional

Question 25. What is the relationship between STUDENT and COURSE entities if:

  • One student can enroll in many courses
  • One course can have many students enrolled
a) One-to-one
b) One-to-many

Question 26. Given: ORDER_DETAIL(OrderID, ProductID, Quantity, ProductName)

Primary Key: {OrderID, ProductID}

If ProductID → ProductName, to achieve 2NF you should:

a) Remove ProductName entirely
b) Create a separate PRODUCT table with ProductID and ProductName
c) Add ProductName to the primary key
d) Leave it as is - it's already in 2NF
ANSWER: b) Create a separate PRODUCT table with ProductID and ProductName - This removes the partial dependency (ProductID → ProductName)

Question 27. Which of the following best describes cardinality constraints?

a) The data types allowed in a column
b) The number of rows in a table
c) The number of entity instances that can participate in a relationship
d) The number of attributes an entity has
ANSWER: c) The number of entity instances that can participate in a relationship - Cardinality defines relationship quantities (1:1, 1:M, M:N)

Question 28. Why is 3NF considered sufficient for most real-world applications?

a) It's the easiest to understand
b) It balances data integrity with reasonable complexity

ANSWER: b) It balances data integrity with reasonable complexity - 3NF provides good integrity without excessive complexity

PART C: SHORT ANSWER (36 points total)

Question 29 (4 points): Explain the difference between a primary key and a foreign key. Provide an example of each.

Primary Key (PK): (2 points)

  • A column (or set of columns) that uniquely identifies each row in a table
  • Cannot contain NULL values
  • Example: StudentID in STUDENT table

Foreign Key (FK): (2 points)

  • A column in one table that references the primary key of another table
  • Creates a relationship/link between tables
  • Example: StudentID in ENROLLMENT table (references STUDENT.StudentID)

Grading: 1 point for correct PK definition, 1 point for PK example, 1 point for correct FK definition, 1 point for FK example

Question 30 (5 points): Consider this unnormalized table:

CardNumber PatronName BookISBN1 BookTitle1 BookISBN2 BookTitle2 CheckoutDate
C001 Alice Brown 978-01 Database Design 978-02 Data Mining 2024-01-15

Identify TWO specific violations of First Normal Form (1NF) in this table schema and explain why each is a problem.

Violation #1: Repeating groups (BookISBN1, BookISBN2, BookTitle1, BookTitle2) (2.5 points)

  • The table has repeating columns for books (Book1, Book2, etc.)
  • This violates 1NF which requires no repeating groups
  • Problems: What if a patron checks out 3 books? Need to add more columns. Wasted space if patron checks out only 1 book. Difficult to query (need to search Book1, Book2, Book3, etc. separately)

Violation #2: The repeating groups create non-atomic structure (2.5 points)

  • The "set of books" is conceptually split across multiple column pairs
  • Each row contains multiple book facts (Book1 data, Book2 data)
  • Proper 1NF would have one book per row
  • Problems: Adding or removing a book requires restructuring. Partial NULL values if fewer than max books checked out

Grading: 1.25 points per violation clearly identified, 1.25 points per explanation of why it's a problem. Must identify TWO distinct 1NF violations related to repeating groups.

Question 32 (7 points): Given this table in 1NF:

a) Transitive dependency / Violates 3NF (2 points)

  • EmpID → DeptID → DeptName, DeptLocation
  • This is a transitive dependency (goes through intermediate non-key attribute)
  • The table is in 2NF (single-column PK, so no partial dependencies possible) but violates 3NF

b) Anomalies that could occur: (2 points - must mention at least one specific anomaly)

  • Update anomaly: If a department changes name or location, must update multiple employee records. Risk of inconsistency if some rows updated but not others.
  • Deletion anomaly: If the last employee in a department is deleted, we lose all information about that department.
  • Insertion anomaly: Cannot add a new department without first hiring an employee in that department.

c) Decomposed tables: (3 points)

EMPLOYEE Table: (1.5 points)

  • EmpID (PK)
  • EmpName
  • DeptID (FK)

DEPARTMENT Table: (1.5 points)

  • DeptID (PK)
  • DeptName
  • DeptLocation

Grading: 1.5 points per table if structure is correct. Deduct 0.25 points per table if PK not clearly marked.

Question 34 (4 points): List the three types of database anomalies that normalization helps prevent and provide a one-sentence description of each.

1. Update Anomaly (~1.33 points)

  • When you have to update the same information in multiple places, creating risk of inconsistency
  • OR: Changing one fact requires updating multiple rows, and you might miss some

2. Insertion Anomaly (~1.33 points)

  • When you cannot add data without also adding other unrelated data
  • OR: Cannot add a new record because other required information is missing or not applicable

3. Deletion Anomaly (~1.34 points)

  • When deleting one piece of information unintentionally causes loss of other important information
  • OR: Removing a record causes loss of other unrelated data that you wanted to keep

Grading: ~1.33 points per anomaly (0.67 for naming, 0.67 for description)

Question 35 (4 points): When might a database designer choose to denormalize a properly normalized database? Give TWO specific scenarios where denormalization would be appropriate and explain WHY denormalization helps in each case.

Grading: 2 points per scenario (1 for scenario description, 1 for explanation of why it helps)

Acceptable scenarios (any TWO):

Scenario #1: Read-heavy reporting/analytics (3 points)

  • Description: Frequently-run reports that join many tables (5+ joins) and perform slowly. Data warehouse or business intelligence applications. Dashboard queries that need sub-second response times.
  • Why it helps: Pre-joining tables eliminates expensive JOIN operations at query time. Reduces number of table lookups needed. Improves query performance significantly. Trade redundancy for speed in read-heavy scenarios.

Scenario #2: Performance-critical real-time queries (3 points)

  • Description: High-traffic web application queries causing bottlenecks. Real-time systems where query speed is critical (e.g., e-commerce product pages). Customer-facing queries that must return in milliseconds.
  • Why it helps: Reduces query complexity and execution time. Decreases database server load. Improves user experience (faster page loads). Can handle higher concurrent user load.

Scenario #3: Calculated/aggregated fields (3 points)

  • Description: Storing pre-calculated totals, counts, or aggregations. Example: Order total stored in ORDER table instead of calculating from ORDER_DETAIL. Expensive calculations that don't change often.
  • Why it helps: Avoids recalculating same value repeatedly. Calculation once during write is cheaper than calculation on every read. Particularly beneficial when calculation is expensive.

Scenario #4: Historical snapshots / Audit trails (3 points)

  • Description: Preserving data as it was at a point in time. Example: Storing customer address on ORDER table (even though also in CUSTOMER table). Audit requirements to preserve exact state at time of transaction.
  • Why it helps: Maintains historical accuracy even if related data changes. If customer moves, old orders still show where they lived when order placed. Required for compliance/audit purposes.

Grading: Must provide TWO distinct scenarios. Generic answers like "for performance" without specifics = 1 point. Must explain WHY denormalization helps.
