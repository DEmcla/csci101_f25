What You MUST Know to Pass

Concept #1: What is a Database Table?

Think of it like a spreadsheet with rows and columns:

  • Rows = individual records (like one student, one order)
  • Columns = properties (like name, date, price)
  • Primary Key (PK) = unique identifier for each row (like Student ID)

Concept #2: Relationships Between Tables

Tables can be connected:

  • One-to-Many (1:M): One customer has many orders
    Implementation: Add foreign key to "many" side
  • Many-to-Many (M:N): Many students take many courses
    Implementation: NEED bridge table (e.g., ENROLLMENT)

Concept #3: The Three Normal Forms (Super Simplified)

  • 1NF: No lists in cells
    ‚ùå Bad: "555-1234, 555-5678"
    ‚úì Good: Separate rows for each phone
  • 2NF: Everything depends on the FULL key (not just part of it)
    Only matters if you have composite key!
  • 3NF: No chains (A‚ÜíB‚ÜíC is bad)
    Most important form - industry standard

Concept #4: Crow's Feet Symbols (MEMORIZE THESE)

|‚Äî  = exactly one (required)        Example: Order ‚Üí Customer
o‚Äî  = zero or one (optional)         Example: Person ‚Üí Driver's License
|‚â∫  = one or more (required, many)   Example: Department ‚Üí Employees
o‚â∫  = zero or more (optional, many)  Example: Customer ‚Üí Orders

Memory Trick:

  • o = "Optional" (starts with O)
  • | = "Mandatory" (stands firm like a wall)
  • ‚Äî = "One"
  • ‚â∫ = "Many" (three toes)

Concept #5: When You Need a Bridge Table

ONLY for Many-to-Many (M:N) relationships!

Example: Students ‚Üî Courses

STUDENT(StudentID, Name)
COURSE(CourseID, Title)
ENROLLMENT(StudentID, CourseID, Grade, Semester)  ‚Üê Bridge table
  PK: {StudentID, CourseID}
  FK: StudentID ‚Üí STUDENT
  FK: CourseID ‚Üí COURSE

Key Rule: One-to-Many does NOT need bridge table!

Intermediate Concepts (Build on Basics)

Keys in More Detail

Primary Key (PK) - Three Types:

  1. Simple PK: One column (e.g., StudentID)
    Most common type
  2. Composite PK: Two+ columns together (e.g., {StudentID, CourseID})
    Common in bridge tables
  3. Surrogate PK: Artificial ID you create (e.g., OrderID = 1, 2, 3...)
    Recommended for most tables

Foreign Key (FK) - Creates Relationships:

  • Points to a Primary Key in another table
  • Can be NULL (if relationship is optional)
  • Can be part of a Primary Key (in bridge tables)

Example:

STUDENT(StudentID, Name)              ‚Üê StudentID is PK
         ‚Üë
         | References
         |
ENROLLMENT(StudentID, CourseID)       ‚Üê StudentID is both FK and part of PK
         PK: {StudentID, CourseID}

Functional Dependencies Explained

Notation: X ‚Üí Y means "if you know X, you can determine Y"

Dependency Example Meaning
StudentID ‚Üí StudentName Know ID 12345 Can look up name "John Smith"
ISBN ‚Üí BookTitle Know ISBN 978-123 Can look up title "Database Design"
{StudentID, CourseID} ‚Üí Grade Know Student 12345 in CS101 Can look up grade "A"

Three Important Terms:

  • Determinant: Left side (X in X ‚Üí Y) - what you know
  • Dependent: Right side (Y in X ‚Üí Y) - what you can find
  • Full Dependency: Depends on the ENTIRE key (good!)

Normal Forms with Examples

Normal Form Rule Example Violation How to Fix
1NF Atomic values Phone: "555-1234, 555-5678" Split into separate rows
2NF No partial dependencies ProductName depends on only ProductID (part of key) Move ProductName to PRODUCT table
3NF No transitive dependencies EmpID ‚Üí DeptID ‚Üí DeptName (chain) Move DeptName to DEPARTMENT table

Basic Questions ‚≠ê (Aim for 100%)

Q1: What shape represents an entity in an ERD?

Answer: Rectangle

Q2: What does "o‚â∫" mean in crow's feet notation?

Answer: Zero or many (optional many)

Q3: Can a primary key be NULL?

Answer: NO - primary keys must have a value

Q4: What does 1NF require?

Answer: Atomic values, no repeating groups

Moderate Questions ‚≠ê‚≠ê (Aim for 70%+)

Q5: Where do you place the FK in a one-to-many relationship?

Answer: In the "many" side table

Q6: If table has single-column PK and is in 1NF, is it automatically in 2NF?

Answer: YES (no composite key = no partial dependencies possible)

Q7: What does 2NF eliminate?

Answer: Partial dependencies (depends on part of composite key)

Q8: What does 3NF eliminate?

Answer: Transitive dependencies (A‚ÜíB‚ÜíC chains)

Challenging Questions ‚≠ê‚≠ê‚≠ê (Aim for 50%+)

Q9: Given PK: {OrderID, ProductID}, is "ProductID ‚Üí ProductName" a partial dependency?

Answer: YES (ProductName depends on only ProductID, not the full key)

Q10: True or False: Partial dependencies violate 3NF.

Answer: FALSE (they violate 2NF, not 3NF)

Q11: Given PK: EmpID, and "EmpID ‚Üí DeptID ‚Üí DeptName", what violates 3NF?

Answer: The transitive dependency (DeptID ‚Üí DeptName)

Q12: Can a table with single-column PK have partial dependencies?

Answer: NO (no "part" of key to depend on - partial only with composite keys)
